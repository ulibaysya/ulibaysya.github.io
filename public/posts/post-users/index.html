POST /users

июля 20, 2026

Внёс изменения в репозиторий, добавляющие handler запроса POST /users. Как несложно догадаться, он используется для добавления нового пользователя.

Доступ к этому запросу должен быть авторизован и доступен только администраторам с соответствующими правами. Обычные пользователи регистрируют аккаунт в основном с помощью входа через внешние сервисы (GET /auth/external/{external}/login/callback), об этом в другом сообщении.

Этот метод создания пользователя не будет использоваться так часто. Он скорее предназначен для особых случаев: выдача каких-то особенных аккаунтов, например для розыгрыша, для ютубера, стримера, контент креатора, influencer’а, для админа.

Также этот метод создания пользователей удобно использовать на ранних этапах разработки, чтобы побыстрому создать каких-то тестовых пустых пользователей.

При добавлении нужно обязательно указать параметр name. Это просто строка, которая обозначает пользователя. На клиенте ее можно отображать на странице профиля пользователя, в header секции, в live drop секции, в розыгрышах, рейтингах. В будущем нужно добавить возможность изменять этот параметр авторизованным пользователям (админам, “владельцу” пользователя (кто-то говорит так вообще? Я имею ввиду самому пользователю)).

Также можно прикрепить параметр image_url. По задумке это должна быть ссылка на аватарку пользователя. Может отображаться там же, где и на клиенте. Если не указать этот параметр, то создастся пользователь с анонимной аватаркой.

По-хорошему, CASEIT должен будет кешировать картинку: скачать ее, обработать (сделать квадратной, сжать), сохранить в S3. Это делается по следующим причинам:

  1. Источником картинки могут являться steam, telegram, другие провайдеры авторизации. Если напрямую указать их картинку, то у пользователей могут быть проблемы с доступностью (блокировки telegram, возможные другие в будущем).

  2. Единый формат автарок (так называемый Anti-Corruption Layer). Условный клиент всегда может быть уверен в том, что аватарки квадратные, что у них примерно одинаковый размер, что они находятся под локальным надзором, что они никуда не пропадут.

Для сохранения аватарки нужно хранилище S3 и сервис по обработке фотографий. От пользователся также нужно будет получить информацию о том, где находятся углы аватарки (допустим, если картинка вертикальная, и условная голова сверху, то квадратное обрезание на нее сместить).

Сейчас image URL сохраняется такой, какая она подаётся. Нужно будет добавить флаг, с помощью которого можно будет обозначить, хотим мы обработу или нет. Стандартное поведение, если флаг отсутсвует - сохранить обработанной. Как и с name, в будущем нужно добавить возможность изменять аватарку авторизованным пользователям.

Также сейчас нет проверки на то, что это действительно URL картинки, что она доступна по HTTPS - нужно добавить.

Также в будущем можно будет добавить возможность загрузить картинку в теле, а не с помощью URL.

Когда картинка будет передаваться от внешнего сервиса (telegram, steam), то нам все равно нужно будет сохранить ее URL, чтобы можно было подгрузить картинку с внешнего сервиса, если пользователь захочет. Telergam, например, выставляет один публичный URL, а потом делает redirect с этого URL на картинку, которая указана как основная. Steam предоставляет API, в котором по steamid64 можно получить аватарку пользователя. По сути ее можно и не сохранять, но я еще не думал об этом, это не главное сейчас.
